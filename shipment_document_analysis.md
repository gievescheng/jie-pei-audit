# 14.3出貨單 可行性分析

更新日期: 2026-03-12

## 結論

可以做，但目前最合理的版本是「自動帶值的出貨單草稿」，不是完全零人工的正式出貨單。

現有資料已足以支撐下列核心欄位:
- 公司名稱
- 訂單編號
- 部分批號 / lot / FOSB 明細
- 計畫出貨日或計畫交期
- 可出貨數量的推導
- 單位

現有資料仍不足以穩定自動帶入下列欄位:
- 申請部門
- 申請人員
- 客戶出貨用產品名稱
- 規格 / 型號
- 備註
- 給客戶看的批號表達方式（例如 `JPAN010601-JPAN010901`）

因此，第一版若要落地，建議做法是:
- 系統先從現有資料自動整理出 `14.3出貨單` 草稿
- 使用者補齊 `申請部門 / 申請人員 / 產品名稱 / 規格 / 備註`
- 批號欄位先提供候選 lot 清單或可勾選 lot，再產生正式檔

## 欄位對照

| 出貨單欄位 | 來源 | 規則 | 狀態 |
| --- | --- | --- | --- |
| 公司名稱 | `14.3出貨單.xlsx` 模板、`inspect_111.json`、`inspect_115.json` | 固定使用 `潔沛企業有限公司` | 可自動帶入 |
| 申請部門 | 無穩定結構來源；樣本檔 row 3 顯示 `資材課` | 可先預設 `資材課`，但這是業務規則，不是資料事實 | 缺資料 |
| 申請人員 | 無穩定結構來源；樣本檔 row 3 顯示 `蔡有為` | 需人工或新增使用者設定 | 缺資料 |
| 日期 | `inspect_111.json` 的預計交期、`verify2.json` 的 `plan_date` | 可用計畫出貨日先帶成草稿；若要正式出貨日，仍應允許人工改 | 可推導 |
| 項次 | 系統生成 | 依列序號 1..n | 可自動帶入 |
| 客戶編號/訂單編號 | `orders_111.json`、`orders_115.json`、`verify2.json.order` | 直接帶入訂單編號 | 可自動帶入 |
| 批號 | `verify2.json.fosb`、`inspect_115.json` row 明細 | 115 資料可帶單一或多個 FOSB/lot；111 樣本中的 `JPAN111001` 未在現有 JSON 明確找到 | 可推導(部分) |
| 產品名稱 | `inspect_111.json` / `inspect_115.json.product` 只有如 `C001/待洗玻璃` | 與樣本出貨單的 `RECYCLE GLASS NEG ABC-1 (JEPE)` 不一致，需產品對照主檔 | 缺資料 |
| 規格/型號 | `inspect_111.json` 的 `12基板...` 可弱推 12 吋；115 資料無穩定規格欄 | 需要產品主檔或欄位對照 | 缺資料 |
| 數量 | `orders_111.json`、`inspect_111.json`、`inspect_115.json`、`verify2.json` | 111 可直接取訂單/計畫數量；115 可依選定 lots 或依 plan date 聚合 good qty / plan qty | 可推導 |
| 單位 | `inspect_111.json`、`inspect_115.json` 皆顯示 `片` | 固定填 `片` | 可自動帶入 |
| 備註 | 無 | 人工輸入 | 缺資料 |

## 與現有樣本的比對

### 樣本 1: `美光20251114`
樣本內容:
- 日期: `2025/11/14`
- 訂單/批號: `4515994888/JPAN111001`
- 產品名稱: `RECYCLE GLASS NEG ABC-1 (JEPE)`
- 規格: `12吋`
- 數量: `200`
- 單位: `片`

目前可對上的欄位:
- `4515994888`: `orders_111.json` 可直接找到
- `2025/11/14`: `inspect_111.json` 中可找到預計交期
- `200片`: `inspect_111.json` 中可找到訂單數量

目前無法直接對上的欄位:
- `JPAN111001`
- `RECYCLE GLASS NEG ABC-1 (JEPE)`
- `12吋`（只有弱訊號，沒有獨立結構欄位）

判斷:
- 2025 樣本可生成半成品草稿
- 若要與現有正式樣本完全一致，仍需補產品主檔與批號對照來源

### 樣本 2: `美光20260112` / `美光20260130`
樣本內容:
- 訂單: `4516243668`
- 批號區間: `JPAN010601-JPAN010901`
- 產品名稱: `RECYCLE GLASS NEG ABC-1 (JEPE)`
- 規格: `12吋`
- 數量: `600`、`1200`

目前可對上的欄位:
- `4516243668`: `orders_115.json`、`verify2.json.order`
- 多個 lot / FOSB: `verify2.json.fosb`、`inspect_115.json` row 明細
- 計畫日期: `verify2.json.plan_date`
- 可累加的生產數量: `verify2.json.actual_qty` / `good`、`inspect_115.json` 的良品數

目前無法直接對上的欄位:
- `JPAN010601-JPAN010901` 這種客戶用批號區間字串
- `RECYCLE GLASS NEG ABC-1 (JEPE)`
- `12吋`

判斷:
- 115 資料的可用性比 111 高，已可支撐 lot 級聚合與草稿出貨單
- 但正式對外名稱與批號表達仍需要 mapping 層

## 建議的第一版生成規則

第一版 `14.3出貨單` 建議採以下規則:
- 先由使用者選訂單編號
- 系統列出該訂單可用的 lot / FOSB 明細
- 使用者勾選本次出貨 lot
- 系統自動聚合數量
- 系統自動帶入公司名稱、日期草稿、訂單編號、批號清單、單位
- 使用者補齊申請部門、申請人員、產品名稱、規格、備註
- 產出 `14.3出貨單.xlsx`

這代表第一版需要的最小新增資料模型是:
- `shipment_request.department`
- `shipment_request.requester`
- `shipment_request.ship_date`
- `shipment_request.remark`
- `product_master(customer_code -> shipping_product_name, spec, unit)`
- `shipment_batch_mapping(order -> display_batch_text)` 或由 lot 清單動態組字串

## 下一批最值得串聯的表單

1. `14 倉儲管理程序/表單/14.3出貨單.xlsx`
   - 直接承接訂單、lot、數量
   - 最接近目前已有資料

2. `12 品質檢驗管理程序/表單/12.5出貨檢查紀錄表.xlsx`
   - 可承接出貨日期、負責單位、檢查項目、出貨批次參考
   - 但檢查結果、執行人、主管覆核仍需人工

3. `14 倉儲管理程序/表單/14.5出貨檢查表.xlsx`
   - 若要把出貨流程上下游串起來，這份表與 12.5 應該一起規劃

## 實作建議

若下一輪要正式開發，不建議直接做「所有程序全部串聯」。

建議順序:
1. 先做 `14.3出貨單` 草稿生成
2. 再做 lot 勾選與數量聚合
3. 再補 `product_master` 與批號 mapping
4. 最後再串 `12.5出貨檢查紀錄表` / `14.5出貨檢查表`

這樣的路徑可以最短時間做出可用版本，而且不會把現在缺失的主檔資料硬寫死在程式裡。
